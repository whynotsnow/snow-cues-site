Snow Cues 可以自部署。自部署的目标是把同一套纯前端应用部署到你信任的 HTTPS 静态站点环境，而不是搭建一个会接收用户秘密的服务端。

如果你只是日常使用，推荐直接使用官方 HTTPS 部署。自部署适合希望控制分发环境、内部固定版本，或在源码 / 构建期扩展算法注册表的用户和维护者。

## 自部署前先确认

自部署应保持这些边界：

- 只分发静态前端产物。
- 不增加账号系统、后端同步或服务端密码库。
- 不接收 `storageData`、空间主密码、关键密钥或密码内容。
- 不通过运行时导入、URL 参数、`storageData` 或远程配置动态加载算法代码。
- 正式使用仍建议选择 Chrome、Edge 等 Chromium 系浏览器。

:::warning
不要把自部署理解为“把用户数据放到自己的服务器”。Snow Cues 的业务数据仍应由用户显式打开、保存、同步和备份。
:::

## 推荐部署方式

推荐部署到 HTTPS 静态托管环境，例如：

- Cloudflare Pages
- Netlify
- Vercel
- GitHub Pages
- 自维护 HTTPS 静态服务

移动端和部分浏览器对 WebCrypto、安全上下文和文件访问能力更敏感。正式使用时不建议用普通局域网 HTTP、移动端 App 内置浏览器或直接打开 `file://` HTML。

## 构建静态产物

在项目目录中安装依赖并构建：

```bash
npm install
npm run build
```

构建产物会输出到 `dist/`。将 `dist/` 作为静态站点根目录部署即可。

部署后应检查：

1. 站点通过 HTTPS 访问。
2. 在 Chrome、Edge 等 Chromium 系浏览器中打开。
3. 应用页面能正常加载。
4. WebCrypto 相关操作没有被浏览器阻断。
5. 新建或打开 `storageData` 的流程能正常完成。

## 本地预览

构建后可以本地预览：

```bash
npm run preview
```

本地开发和预览可使用 `localhost` 或 `127.0.0.1`。这不等同于正式部署；正式部署仍建议使用 HTTPS。

## 部署后如何使用

自部署站点和官方站点一样，只负责分发前端应用。

用户仍然需要自己管理：

- `storageData`
- 空间主密码
- 单条密码的关键密钥
- 导出备份
- Syncthing 等外部同步工具

自部署不会提供主密码找回、关键密钥找回、云端同步或自动合并冲突。

## 更新自部署版本

更新自部署版本前，建议：

1. 先备份现有 `storageData`。
2. 阅读版本说明，尤其是规则链、导入 Rule、`params` 或存储格式相关变化。
3. 在测试空间中验证新版本能正常进入空间、查看条目和保存数据。
4. 再替换正式静态产物。

如果某个空间规则使用了 `params`，不要再用不支持 `params` 的旧版本打开和维护该空间。

## 自部署算法扩展

官方构建只包含项目维护者审计过的规则算法模板。自部署版本可以在源码 / 构建期扩展算法注册表并自行构建。

这个能力适合：

- 组织内部固定使用一组经过内部审计的派生算法模板。
- 个人 fork 后加入官方版本暂未提供的 WebCrypto 或 WASM-backed 算法。
- 在不修改业务数据主体模型的前提下，让声明式导入规则选择更多源码内置模板。

不支持：

- 让用户在页面中粘贴 JavaScript、表达式、WASM URL 或远程脚本。
- 让 `storageData` 携带算法代码。
- 通过 URL 参数决定运行哪个算法实现。
- 通过远程规则市场或第三方 JSON 动态下发算法。

自部署构建和官方构建可能使用不同算法注册表。使用自部署专有算法创建的空间，必须继续使用包含该算法的构建打开；官方构建无法使用未注册算法维护该空间。

## 自部署维护责任

自部署方需要自己负责：

- 静态站点托管和 HTTPS 配置。
- 构建产物的来源可信度。
- 浏览器兼容性验证。
- 自部署专有算法的安全性、性能和长期维护。
- 向使用者说明当前构建包含哪些算法模板。
- 在升级前验证旧 `storageData` 和规则链兼容性。
